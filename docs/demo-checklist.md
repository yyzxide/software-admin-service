# 本地演示和验收清单

## 演示目标

演示时不要从代码细节开始，先证明这个后台能跑通完整业务闭环：

```text
登录 -> 查看权限配置 -> 上传软件 -> 查看列表和详情 -> 分片上传设计说明 -> 提交审核 -> 审核通过 -> 上架 -> 查看操作日志 -> 查看 Swagger 文档
```

## 启动顺序

```bash
cd java-software-admin-service
cp .env.example .env
make doctor
make docker-up
make init-db
make test
make build
make run
```

服务启动后打开：

```text
管理页面：http://127.0.0.1:8090/admin/index.html
接口文档：http://127.0.0.1:8090/swagger-ui.html
OpenAPI JSON：http://127.0.0.1:8090/v3/api-docs
```

默认账号：

```text
admin / admin123456
```

## 一键验收

服务运行后执行：

```bash
make smoke
```

验收内容：

- 健康接口可访问。
- 管理员可以登录。
- Token 可以访问 `/auth/me`。
- OpenAPI 文档包含核心路径。
- 分类、标签、软件列表、审核任务、操作日志接口可访问。

## 页面演示步骤

1. 登录后台。
2. 到“权限管理”，展示账号、角色和默认收起的权限字典。
3. 打开“接口文档”，展示 Swagger UI。
4. 回到“软件管理”，查询软件列表。
5. 上传一个 `.deb`、`.rpm` 或 `.appimage` 测试包。
6. 打开 `docs/upload-resume-and-signature.md`，说明大文件分片上传、断点续传和签名校验。
7. 查看软件详情。
8. 点击“提审”。
9. 到“审核中心”，刷新审核任务。
10. 分配审核人。
11. 审核通过。
12. 回到“软件管理”，确认状态变化。
13. 到“操作日志”，查询操作记录。
14. 到“联调检查”，执行接口检查。

## 后端讲解顺序

推荐讲法：

1. 业务背景：企业软件商店后台，需要管理软件包从上传到上架。
2. 数据模型：软件、版本、安装包、分类、标签、审核任务、操作日志。
3. 上传事务：普通上传、分片上传、算 SHA256、验签、多表写入、缓存失效、操作日志。
4. 审核状态机：待审核、审核中、已通过、已驳回。
5. 安全控制：Token、上传大小、格式校验、路径约束、断点续传、签名校验、操作审计。
6. 工程质量：RBAC 权限、单元测试、Swagger 文档、冒烟脚本、Docker 本地依赖和后端容器化启动。

## 常见问题准备

### 为什么是模块化单体？

研发阶段优先保证业务闭环和本地联调效率。模块边界在包、Service 和表结构上保持清晰，后续可以把存储、审核、通知拆出去。

### 上传失败怎么处理？

当前数据库写入在事务内，文件保存先落本地。生产环境需要补偿清理策略，例如数据库失败后删除已保存文件，或通过异步任务清理孤儿文件。

### 为什么需要操作日志？

后台管理系统必须可追溯。软件上传、编辑、上下架、审核动作都要记录操作人、资源、动作、时间和详情，便于审计和问题排查。

### 这个项目还缺什么？

当前已经具备本地演示和面试讲解所需的核心闭环。后续重点是生产级密码策略、菜单权限隐藏、安装包安全扫描执行器、对象存储、正式包孤儿文件清理和生产部署。
