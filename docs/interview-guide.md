# 面试讲解稿

## 一句话介绍

这是一个基于 Java / Spring Boot 的企业软件商店后台管理系统，核心解决软件包从上传、版本维护、审核到上架发布的管理流程，并提供操作审计和本地验证能力。

## 项目背景

企业内部软件分发场景里，软件包不是简单上传文件就结束。后台需要管理软件基本信息、系统和 CPU 架构适配、版本、安装包、分类标签、审核状态和上下架记录。这个项目围绕这条链路做了一个可本地运行、可测试的 Java 后台系统。

## 我的主要工作

可以这样讲：

- 负责后台核心数据模型设计，包括软件、版本、安装包、分类、标签、审核任务和操作日志。
- 实现软件上传和版本管理，多表事务写入保证元数据一致。
- 实现审核工作流，支持提交审核、分配审核人、通过、驳回和历史记录。
- 实现操作审计，将软件、版本、安装包和审核动作记录到日志表。
- 补充上传安全控制，包括单文件大小限制、总请求大小限制、安装包格式和后缀一致性校验。
- 设计并实现分片上传和断点续传，支持上传会话、分片进度、合并校验、SHA256 完整性校验和可选 RSA 签名验签。
- 接入 Redis 缓存，针对分类、标签和软件详情做读缓存与写失效。
- 补充单元测试，覆盖状态流、参数校验和主要业务规则。

## 可以重点讲的技术点

### 1. 为什么采用模块化单体

研发阶段更重视完整闭环和低成本联调。拆成多个服务会增加本地启动、事务一致性和排查成本。因此当前把模块边界放在包结构、Service 接口和数据库表上，后续可以按审核、存储、通知继续拆分。

### 2. 软件上传如何保证一致性

上传接口会校验分类、标签、系统、架构和包格式。小文件可以直接 multipart 上传，大文件先创建上传会话并按分片上传，合并后计算 SHA256 和签名状态。数据库写入在一个事务中完成：`apps`、`app_versions`、`app_packages`、`app_tags` 一起成功或失败。写入完成后清理软件详情缓存，并写操作日志。

### 3. 审核流怎么设计

审核任务表 `review_tasks` 记录当前状态，审核历史表 `review_histories` 记录每次流转。软件状态和版本状态会随审核动作变化：提交后进入审核中，通过后上架，驳回后进入审核驳回。

### 4. 操作日志为什么内置在 admin-service

当前阶段需要保证业务动作和审计记录尽量处于同一事务语义里。如果把日志服务拆出去，失败补偿和消息一致性会复杂很多。所以研发阶段先内置，后续再根据吞吐和隔离要求拆分。

### 5. 上传安全怎么做

上传有三层限制。第一层是 Spring Multipart 的 `max-file-size` 和 `max-request-size`，在进入业务之前拦住过大的请求。第二层是业务层校验安装包不能为空、不能超过业务上限、`packageFormat` 必须和文件后缀一致。第三层是存储层净化文件名、限制保存路径在包目录内，并计算 SHA256。

大文件上传会先创建 `package_upload_sessions`，前端按 `chunkIndex` 上传分片；断网后查询 `uploaded_chunks`，只补传缺失分片。所有分片到齐后后端按顺序合并，校验文件大小和 SHA256，如果配置 `sha256-rsa` 则用公钥验签。业务接口只消费已完成的 `uploadSessionId`，上传基础设施和软件业务解耦。

如果未来支持一次上传多个文件，我会在业务层把多个 `MultipartFile.getSize()` 累加，超过批次总大小就拒绝整个请求，避免多个小文件绕过单文件限制。

### 6. MyBatis XML 的选择

后台管理系统有较多筛选、排序、统计和多表聚合。XML SQL 更容易控制查询结构和索引命中，也便于对复杂 SQL 做审查。

## 面试可能被问

### Q1：这个项目和普通 CRUD 后台有什么区别？

不是单纯 CRUD。重点是软件、版本、安装包、审核任务、操作日志之间的状态联动。比如审核通过不只是改一张表，而是要更新任务、历史、软件状态、版本状态和审计日志。

### Q2：如果审核通过时操作日志写失败怎么办？

当前实现里日志和业务在同一服务中执行，数据库异常会导致事务失败，避免出现业务成功但没有审计记录。后续如果日志量很大，可以改成事务内写 outbox，再异步投递日志事件。

### Q3：为什么没有一开始做微服务？

项目处于研发和验证阶段，单体更容易保证闭环、降低联调成本。现在包结构已经按领域拆分，后续可以把存储、审核、通知拆出服务。

### Q4：怎么优化上传？

当前已经支持普通 Multipart、分片上传和过期上传会话清理。后续优化重点是正式包孤儿文件清理、对象存储和病毒扫描执行器。

### Q5：怎么做权限？

当前采用 RBAC：`admin_users`、`admin_roles`、`admin_permissions`、`admin_user_roles`、`admin_role_permissions`。请求先经过 Token 鉴权，再由拦截器读取 Controller 上的 `@RequirePermission`，根据当前管理员角色权限判断是否允许访问。默认 `admin` 绑定 `super_admin`，拥有 `*` 通配权限。后台提供权限管理页面，可以创建运营、审核、只读等账号，给用户分配角色，再给角色分配权限点。

### Q6：多个文件总大小过大怎么办？

分三层处理：网关限制请求体总大小，Spring Multipart 限制 `max-request-size`，业务层遍历所有文件累加大小。如果总大小超过批次上限，就在保存文件和写数据库之前整体拒绝。当前项目一次只上传一个 `packageFile`，所以主要靠单文件上限和总请求上限保护。
