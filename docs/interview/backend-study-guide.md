# 后端面试知识地图

这份文档不是八股目录的机械堆砌，而是从本项目的真实代码反推面试必须掌握的 Java 后端知识。目标是做到三层回答：先说结论，再解释原理，最后能指到项目代码、数据库表和测试用例。

## 先定标准

每个知识点至少做到：

1. 能用自己的话解释，不背源码定义。
2. 能说明项目为什么使用它，以及不用会出现什么问题。
3. 能指出实现位置、失败场景和仍未解决的生产边界。

项目主线建议固定为：

> 软件与安装包上传 -> 版本和包元数据写入 -> 安全状态校验 -> 提交审核 -> 并发受控的审核决定 -> 上下架 -> 缓存失效和操作审计。

## 一、Java 基础

### 必须掌握

- 面向对象：封装、继承、多态，接口与抽象类的区别。
- 集合：`List`、`Set`、`Map` 的使用场景，`HashMap` 的散列、扩容和冲突处理。
- `equals` / `hashCode` 契约，以及对象作为 Map key 时的风险。
- 泛型、类型擦除、通配符 `? extends` / `? super`。
- checked exception 与 runtime exception，异常转换和异常链。
- `String` 不可变性，`StringBuilder` 的适用场景。
- `LocalDateTime`、时区和时间边界；为什么日期末尾不应简单写成 `23:59:59`。
- Java IO / NIO：流、Path、Files、原子移动、临时文件、资源关闭。
- 并发基础：线程安全、可见性、原子性、有序性，`synchronized`、锁和线程池的基本原理。

### 项目对应

- `PackageFileStorageService`：`Path.normalize()`、根目录约束、临时文件和原子移动。
- `PackageUploadSessionService`：文件 IO 失败转换、补偿清理、分片顺序合并。
- `OperationLogServiceImpl`：日期解析、结束时间转为右开区间 `[start, end)`。
- 状态枚举：`SoftwareStatus`、`AppVersionStatus`、`ReviewTaskStatus`，避免魔法数字散落。

### 高频追问

- `HashMap` 为什么线程不安全？这里为什么没有用它保存线上状态？
- `finally`、try-with-resources 和资源泄漏分别怎么理解？
- SHA256 是加密吗？不是，它是单向摘要；RSA 验签验证的是签名，不是解密安装包。

## 二、Spring 与 Spring MVC

### 必须掌握

- IoC / DI 解决什么问题，构造器注入为什么更适合测试。
- Bean 生命周期、单例 Bean 的线程安全边界。
- `@Component`、`@Service`、`@Repository`、`@Configuration` 的职责。
- Spring MVC 请求链：Filter -> Interceptor -> Controller -> Service -> Mapper。
- `@RequestBody`、`@ModelAttribute`、Multipart 参数绑定的区别。
- Bean Validation：`@Valid`、字段约束和全局异常转换。
- AOP 和代理：JDK 动态代理、CGLIB 的基本区别。
- 配置加载：`application.yml`、环境变量、profile 和敏感配置外置。

### 项目对应

- `AdminAuthInterceptor`：Token 校验和权限拦截位于 Controller 之前。
- `@RequirePermission`：方法或类级权限声明。
- `GlobalExceptionHandler`：参数错误、业务异常、数据库唯一键冲突的统一响应。
- `SoftwareControllerTest`：验证 Multipart 参数绑定和请求校验。

### 必须能答

- 为什么鉴权放 Interceptor，而业务权限仍可能在 Service 二次校验？

  Interceptor 解决“调用者是否拥有接口权限”，Service 解决“这个具体资源是否归属于调用者、目标账号是否具备业务资格”。例如有 `review:approve` 权限，不代表可以处理已经分配给别人的任务。

- 为什么 DTO 同时有 camelCase 和 snake_case setter？

  JSON 可以由 Jackson 命名策略处理，但查询参数和 Multipart 由 Spring 数据绑定器处理，不能盲目删除兼容 setter。

## 三、Spring 事务

### 必须掌握

- ACID：原子性、一致性、隔离性、持久性。
- MySQL InnoDB 的四种隔离级别，以及脏读、不可重复读、幻读。
- Spring `@Transactional` 基于代理，类内自调用为什么可能失效。
- 默认回滚规则：通常只对 `RuntimeException` / `Error` 回滚。
- 传播行为：至少理解 `REQUIRED`、`REQUIRES_NEW`。
- 数据库事务不能自动回滚本地文件、远程调用和缓存。
- `afterCommit` 的意义和限制。

### 项目对应

- 软件上传：`apps`、`app_versions`、`app_packages`、`app_tags` 同事务写入。
- 审核决定：任务状态、审核历史、软件/版本状态同事务更新。
- `TransactionActions`：事务提交后再失效缓存或删除确认过期的分片目录。
- `OperationLogEventListener`：`AFTER_COMMIT` 写审计，异常只记录日志。

### 关键边界

数据库事务成功、文件写入失败，或文件成功、数据库回滚，都会产生双写问题。当前项目使用失败补偿删除降低孤儿文件风险，但它不是分布式事务。生产化可以增加对象存储临时区、状态表、对账巡检和可重试任务。

操作日志当前也是提交后尽力写入。若审计必须不丢，应在业务事务内写 Outbox 表，再异步消费、重试和告警。

## 四、MySQL 与索引

### 必须掌握

- InnoDB 聚簇索引、二级索引、回表、覆盖索引。
- B+ Tree 为什么适合范围查询。
- 联合索引最左前缀，索引下推，选择性。
- `EXPLAIN` 关注 `type`、`key`、`rows`、`Extra`。
- 唯一索引既是性能结构，也是并发下的最终一致性约束。
- 行锁、间隙锁、死锁原因与重试策略。
- `SELECT ... FOR UPDATE` 的生效条件：必须在事务中，并命中合适索引。
- 深分页问题和基于游标/主键的翻页方案。
- `DATETIME(3)` 的毫秒精度与时间范围查询。

### 项目对应

- `uk_app_key`：软件业务标识唯一。
- `uk_app_version_code`：同一软件版本号唯一。
- `uk_version_os_arch_format`：同一版本安装包变体唯一。
- `uk_review_active_target`：生成列 `active_review_key` 只约束活动审核任务。
- 上传会话行锁：串行保护 `uploaded_chunks` 和更新时间。
- 操作日志：按 `created_at DESC, id DESC` 排序，日期结束值使用下一时刻并配合 `<`。

### 面试要讲清

“先查重再插入”不能解决并发竞态，因为两个事务都可能查到 0。正确做法是数据库唯一键兜底，并把 `DuplicateKeyException` 转为业务冲突；前置查询只用于更友好的错误提示。

## 五、MyBatis

### 必须掌握

- Mapper 接口与 XML 的绑定方式。
- `#{}` 使用预编译参数，`${}` 是文本拼接，后者容易产生 SQL 注入。
- `resultMap`、动态 `<where>` / `<if>` / `<choose>`。
- affected rows 如何用于并发状态判断。
- N+1 查询、批量查询和大结果集问题。
- SQL 仍应由数据库约束和索引支撑，不能把所有规则只写在 Java。

### 项目对应

- `ReviewMapper.xml`：活动任务查询、审核人资格查询、条件分配和条件完成。
- `PackageUploadSessionMapper.xml`：`FOR UPDATE` 和条件过期更新。
- `OperationLogMapper.xml`：动态筛选、统计和右开时间范围。

MyBatis XML 本身不是简历亮点。真正值得讲的是你如何用 SQL 条件、唯一键、行锁和 affected rows 解决竞态。

## 六、并发、幂等与状态机

### 四种手段要分清

1. 唯一键：阻止重复资源，例如同一审核目标存在两条活动任务。
2. 条件更新：`UPDATE ... WHERE status IN (...)`，只有旧状态符合才成功。
3. 悲观锁：`SELECT ... FOR UPDATE`，适合短事务内串行修改同一行。
4. 幂等语义：重复请求的最终状态相同，并且不制造额外副作用。

### 项目案例

- 重复上架/下架直接返回当前状态，不刷新发布时间、不重复记录状态变更。
- 同一分片重传必须原子覆盖，不能因为大小相同就跳过；相同大小不代表相同内容。
- 完成上传前先进入 `MERGING`，避免重复合并。
- 过期清理必须再次检查状态和 `updated_at`，否则可能删除刚恢复的上传。
- 审核任务被分配后只允许归属审核人决定。
- 版本级驳回只改版本，不改已上架软件。

### 必须会画

至少能在纸上画出软件状态、版本状态、审核任务状态和上传会话状态，并说明哪些转换由 Service 校验、哪些由 SQL 条件或唯一键兜底。

## 七、Redis 与缓存一致性

### 必须掌握

- Cache Aside：读缓存未命中查数据库并回填，写数据库后删除缓存。
- 为什么通常删除缓存而不是直接更新缓存。
- 为什么缓存失效应放在事务提交后。
- 缓存穿透、击穿、雪崩、热 key、大 key 的处理思路。
- TTL、随机过期、空值缓存、互斥重建的适用场景。
- Redis 不可用时核心业务是否应该降级。

### 项目对应

分类、标签、软件详情是读多写少数据。Cache Service 捕获 Redis 异常并回退数据库；写操作注册提交后失效，避免数据库回滚但缓存已提前删除。当前没有解决极端并发下“旧读回填覆盖删除”的全部问题，生产可以使用延迟双删、版本号或消息驱动失效。

## 八、认证、授权与安全

### 必须掌握

- 认证 Authentication 与授权 Authorization 的区别。
- JWT 的 Header、Payload、Signature 结构，标准 Claims 与自定义 Claims 的区别。
- HS256 本质上使用 HMAC，需掌握完整性校验、密钥长度、泄露后果和轮换方式，以及与 RS256 的差异。
- BCrypt 自带盐且计算成本可调，为什么优于直接 SHA256 存密码。
- Token 过期、会话版本、账号禁用和旧 Token 失效。
- RBAC：用户 -> 角色 -> 权限点；通配权限匹配。
- 401 与 403 的区别。
- 路径穿越、恶意文件名、超大请求、扩展名伪造。
- SHA256 完整性、RSA 签名真实性与杀毒扫描解决的是不同问题。

### 项目边界

- 当前 JWT 使用 HS256，密钥至少 32 字节；非生产 profile 有开发默认值，只适合本地研发，生产必须通过环境变量或密钥系统注入并禁止默认值。
- 当前未接真实杀毒引擎，只支持人工/外部流程更新安全状态，并在审核和上架前阻断非 SAFE 包。
- 当前没有登录失败限流、MFA、密钥轮换和细粒度数据权限，不能包装成完整 IAM。

## 九、文件上传与存储

### 必须掌握

- Multipart 请求在 Web 容器、Spring 和业务层的大小限制。
- 分片大小、总分片数、最后一片大小的计算。
- 断点续传如何查询已上传分片。
- 临时文件 + 原子移动避免半写文件。
- SHA256 校验为什么应在完整合并后再次执行。
- 文件系统与数据库一致性、孤儿文件和定时对账。
- 本地目录、NFS、MinIO/S3 的适用场景和差异。

### 当前选择

内部研发测试部署在虚拟机，本地目录成本最低且足够验证流程，因此不是当前缺陷。它的生产限制是多实例共享、容量扩展、备份、权限和生命周期管理；达到这些需求再抽象对象存储更合理。

## 十一、测试

### 必须掌握

- 单元测试：隔离 Service 规则，速度快，但不能证明 SQL 和数据库行为。
- Controller 测试：验证路由、参数绑定、Validation 和响应。
- 集成测试：真实 Spring Context、MySQL、Redis、文件系统共同运行。
- Fake 与 Mock 的区别；过度 Mock 会让测试只验证自己的假实现。
- 回归测试应对应已经发现的 bug，而不只覆盖 happy path。

### 当前结果

最近一次 `mvn test`：89 个测试、0 失败、1 个跳过。跳过的是 Testcontainers MySQL 用例，因为执行机器的 Docker API 与 Testcontainers 不兼容。面试时必须如实说“单元与 Controller 测试已通过，真实 MySQL 集成项本机跳过”，不能说全部集成测试通过。

### 重点回归用例

- 版本驳回不污染已上架软件。
- 无包、不可用包、风险包不能发布。
- 同大小损坏分片可以被重传覆盖。
- 恢复上传后，过期清理不会误删目录。
- 已分配任务不能被其他审核人处理。
- 审计写入失败不让已提交业务返回失败。
- 日期查询包含结束日最后一秒的毫秒数据。

### 必须掌握

- Compose 适合本地依赖编排，不等于生产编排。

## 十三、架构取舍

### 为什么是模块化单体

- 当前团队和业务规模更需要快速联调与本地验证。
- 单库事务可以直接保证核心多表一致性。
- 包、Service 和表已经按领域拆分，后续有真实独立扩缩容或团队边界时再拆服务。

### 什么时候拆微服务

只有出现独立扩缩容、不同发布节奏、明确团队边界、故障隔离或数据所有权需求时再拆。拆分后必须面对分布式事务、消息幂等、服务治理、链路追踪和运维成本。

### 三个生产化方案

1. 存储：本地目录 -> `ObjectStorage` 接口 -> MinIO/S3，增加临时区和生命周期。
2. 安全：人工状态 -> 异步扫描任务 -> 回调验签、幂等更新、超时重试。
3. 审计：`AFTER_COMMIT` 尽力写 -> 事务 Outbox -> 消费重试、死信和告警。

## 最后提醒

这个项目已经足够作为一年经验候选人的后端主项目，但“代码存在”不等于“你掌握了”。面试官真正区分 AI 生成项目和本人项目的方法，就是连续追问状态边界、并发时序、事务失效、SQL 约束和失败补偿。你能从需求讲到代码、从代码讲到测试、再主动说出生产边界，项目才真正属于你。
