# 面试讲解稿

## 一句话介绍

这是一个基于 Java / Spring Boot 的企业软件商店后台管理系统，核心解决软件包从上传、版本维护、审核到上架发布的管理流程，并提供操作审计和本地验证能力。

## 项目背景

企业内部软件分发场景里，软件包不是简单上传文件就结束。后台需要管理软件基本信息、系统和 CPU 架构适配、版本、安装包、分类标签、审核状态和上下架记录。这个项目围绕这条链路做了一个可本地运行、可测试的 Java 后台系统。

## 我的主要工作

建议按“参与内部需求和方案梳理，后续负责 Java 后台实现与完善”的口径讲，避免说成生产系统总负责人：

- 负责后台核心数据模型设计，包括软件、版本、安装包、分类、标签、审核任务和操作日志。
- 实现软件上传和版本管理，多表事务写入保证元数据一致。
- 实现审核工作流，支持提交、分配、通过、驳回和历史记录，并补充活动任务唯一约束、审核人资格/归属和版本级状态隔离。
- 实现操作审计，将软件、版本、安装包和审核动作记录到日志表。
- 补充上传安全控制，包括单文件大小限制、总请求大小限制、安装包格式和后缀一致性校验。
- 设计并实现分片上传和断点续传，支持上传会话、分片进度、合并校验、SHA256 完整性校验和可选 RSA 签名验签。
- 接入 Redis 缓存，针对分类、标签和软件详情做读缓存与写失效。
- 补充单元测试，覆盖状态流、上传竞态、权限归属、日期边界、参数校验和主要业务规则。

## 可以重点讲的技术点

### 1. 为什么采用模块化单体

研发阶段更重视完整闭环和低成本联调。拆成多个服务会增加本地启动、事务一致性和排查成本。因此当前把模块边界放在包结构、Service 接口和数据库表上，后续可以按审核、存储、通知继续拆分。

### 2. 软件上传如何保证一致性

上传接口会校验分类、标签、系统、架构和包格式。小文件可以直接 multipart 上传，大文件先创建上传会话并按分片上传，合并后计算 SHA256 和签名状态。数据库写入在一个事务中完成：`apps`、`app_versions`、`app_packages`、`app_tags` 一起成功或失败。缓存失效和审计事件都注册在事务提交之后，回滚时不执行。

### 3. 审核流怎么设计

审核任务表 `review_tasks` 记录当前状态，审核历史表 `review_histories` 记录每次流转。软件级提审会更新软件和草稿版本；版本级提审只更新目标版本，驳回也只驳回该版本，避免影响已经上架的软件。数据库生成列和唯一键限制同一目标只有一个活动任务，决定动作通过条件更新判断是否被并发处理。

### 4. 操作日志为什么内置在 admin-service

当前阶段把审计模块内置在同一个应用中，降低部署和排查成本。业务方法发布事务事件，监听器在 `AFTER_COMMIT` 写日志：业务回滚时不写误导日志，日志写失败时只记录错误，不让已经提交的业务请求返回假失败。它是尽力审计，不是强一致；生产强审计应在业务事务内写 Outbox 并重试消费。

### 5. 上传安全怎么做

上传有三层限制。第一层是 Spring Multipart 的 `max-file-size` 和 `max-request-size`，在进入业务之前拦住过大的请求。第二层是业务层校验安装包不能为空、不能超过业务上限、`packageFormat` 必须和文件后缀一致。第三层是存储层净化文件名、限制保存路径在包目录内，并计算 SHA256。

大文件上传会先创建 `package_upload_sessions`，前端按 `chunkIndex` 上传分片；断网后查询 `uploaded_chunks`，只补传缺失分片。上传分片时用 `SELECT ... FOR UPDATE` 锁定会话行，重复分片重新写临时文件后原子替换，不能只凭大小判断内容相同。所有分片到齐后按顺序合并并校验大小和 SHA256，如果配置 `sha256-rsa` 则用公钥验签。业务接口只消费已完成的 `uploadSessionId`，上传基础设施和软件业务解耦。

如果未来支持一次上传多个文件，我会在业务层把多个 `MultipartFile.getSize()` 累加，超过批次总大小就拒绝整个请求，避免多个小文件绕过单文件限制。

### 6. MyBatis XML 的选择

后台管理系统有较多筛选、排序、统计和多表聚合。XML SQL 更容易控制查询结构和索引命中，也便于对复杂 SQL 做审查。

## 面试可能被问

### Q1：这个项目和普通 CRUD 后台有什么区别？

不是单纯 CRUD。重点是软件、版本、安装包、审核任务、操作日志之间的状态联动。比如审核通过不只是改一张表，而是要更新任务、历史、软件状态、版本状态和审计日志。

### Q2：如果审核通过时操作日志写失败怎么办？

当前日志监听器在业务事务提交后执行。这样业务回滚不会留下误导日志，日志数据库异常也不会让已经提交的业务返回失败；代价是异常时可能出现审计缺口，当前会记录错误日志。若要求日志不可丢，方案是业务事务内写 Outbox，再由后台任务带重试消费，而不是让 `AFTER_COMMIT` 监听器反向回滚已经提交的事务。

### Q3：为什么没有一开始做微服务？

项目处于研发和验证阶段，单体更容易保证闭环、降低联调成本。现在包结构已经按领域拆分，后续可以把存储、审核、通知拆出服务。

### Q4：怎么优化上传？

当前已经支持普通 Multipart、分片上传、分片原子覆盖、会话行锁、完成合并保护、条件过期清理、失败落盘文件清理和安装包安全状态拦截。后续生产化重点是孤儿文件巡检、对象存储扩展和真实安全扫描执行器。

### Q5：怎么做权限？

当前采用 RBAC：`admin_users`、`admin_roles`、`admin_permissions`、`admin_user_roles`、`admin_role_permissions`。请求先经过 Token 鉴权，再由拦截器读取 Controller 上的 `@RequirePermission`。管理员密码使用 BCrypt，Token 带会话版本，账号禁用或密码重置后旧 Token 失效。审核分配还做业务层二次校验：目标账号必须启用且同时具备通过、驳回权限；任务分配后只能由归属审核人处理。

### Q6：多个文件总大小过大怎么办？

分三层处理：网关限制请求体总大小，Spring Multipart 限制 `max-request-size`，业务层遍历所有文件累加大小。如果总大小超过批次上限，就在保存文件和写数据库之前整体拒绝。当前项目一次只上传一个 `packageFile`，所以主要靠单文件上限和总请求上限保护。

### Q7：为什么版本审核不能直接复用软件审核状态？

因为软件可能已经在线，只是在提交新版本。如果版本提审把软件改成 `REVIEWING`，或者版本驳回把软件改成 `REJECTED`，会错误影响现有在线版本。当前实现把目标分开：软件级任务联动软件和草稿版本；版本级任务只修改目标版本，通过时再更新 latest 和发布状态。

### Q8：如何防止审核和上传并发覆盖？

审核有三层：数据库唯一键防重复活动任务，分配和完成使用带旧状态条件的 `UPDATE`，再根据 affected rows 判断是否被别人处理。上传分片用会话行锁串行化进度更新，完成合并先条件切换到 `MERGING`，过期清理也必须在状态和时间条件仍成立时才能标记失败。

### Q9：为什么本地目录和手动安全状态现在是合理的？

项目定位是内部研发测试，部署在虚拟机，本地目录可以降低依赖和联调成本；真实扫描引擎不属于当前已有能力，所以只实现安全状态闭环和发布拦截，不能说已经接入杀毒。生产化时再把存储抽象到 MinIO/S3，并接异步扫描回调。

### Q10：怎么证明测试不是只测 happy path？

最近一次执行是 83 个测试、0 失败、1 个因无 Docker 跳过。除了正常上传发布，还覆盖损坏分片重传、清理与恢复竞态、无安装包发布、风险包拦截、重复提审、并发决定、版本驳回隔离、审核人归属、提交后审计失败和日期/MySQL 时间范围边界。真实 MySQL 约束仍要在 Docker 可用环境运行 Testcontainers，不能把跳过项说成已通过。

### Q11：`@Transactional` 最容易被追问什么？

要能解释 Spring 代理、自调用失效、默认只对运行时异常回滚、事务传播和隔离级别。本项目还要说清数据库事务只能直接约束数据库写入，文件系统不参与事务，所以失败路径需要补偿删除；缓存和日志使用提交后回调，避免回滚后产生错误副作用。

### Q12：你认为这个项目目前最大的生产差距是什么？

不是“缺少更多 CRUD 页面”，而是真实扫描、对象存储、强审计 Outbox、完整 MySQL/Redis/文件集成测试、监控告警和生产密钥管理。面试时主动说出这些边界，比声称生产级更可信。
