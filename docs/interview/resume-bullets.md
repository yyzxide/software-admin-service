# 简历写法和面试表达

## 项目名称

信创软件商店后台管理系统

## 一句话描述

基于 Java / Spring Boot 实现企业软件商店后台管理服务，覆盖软件上传、版本管理、安装包管理、审核流、上下架、操作审计和本地可视化验证。

## 面试开场

可以这样说：

> 这个项目是企业软件商店后台管理服务，主要解决软件包从上传、版本维护、审核到上架发布的后台管理链路。我负责的是 Java 后台核心模块，包括软件、版本、安装包、分类标签、审核任务和操作日志。项目用 Spring Boot + MyBatis + MySQL + Redis 实现，重点不是普通 CRUD，而是多表事务、状态流转、缓存失效、上传安全和审计追踪。

## 重点讲解点

### 1. 业务闭环

从软件上传开始，系统会保存安装包、计算 SHA256、写入软件主表、版本表、安装包表、标签关联表，然后进入审核和上下架流程，最后所有关键动作写操作日志。

### 2. 多表事务

上传不是只保存一个文件，还要写多个元数据表。如果其中任何一步失败，数据库事务会回滚，避免出现有软件没有版本、有版本没有安装包的脏数据。

### 3. 审核状态机

审核任务和业务状态分开建模。`review_tasks` 记录审核任务状态，`review_histories` 记录流转历史，软件和版本表记录最终业务状态。

### 4. 操作审计

后台系统要能追责和排查问题，所以软件上传、编辑、上架、下架、审核通过、审核驳回都会记录操作日志。

### 5. 上传安全

当前做了单文件大小、总请求大小、安装包格式和后缀一致性校验，并对文件名和保存路径做约束。大文件走上传会话和分片进度，重复分片按幂等处理，完成合并前会进入“合并中”状态防止并发完成；合并后做 SHA256 校验，也支持 `sha256-rsa` 公钥验签。过期上传会话会定时清理，失败路径会尽量清理已落盘正式包，审核通过和上架前会拦截签名失败、未标记安全、处理失败或安全状态有风险的安装包。

### 6. 工程化

项目有 Swagger UI、README、接口文档、演示清单、单元测试、冒烟脚本和 Docker Compose 本地依赖，也支持通过 Docker profile 一键启动后端。

## 表达边界

不建议夸大项目背景或上线状态，例如：

- “这是已经承载生产流量的核心系统。”
- “这是完整生产级 IAM / 制品安全平台。”
- “所有能力都已经达到线上发布标准。”

建议说：

- “这是我参与/复盘企业软件商店管理后台后，独立整理并实现的一套 Java 后台服务。”
- “项目定位是研发和内测阶段后台能力验证。”
- “重点是把业务闭环、状态流、事务一致性和审计能力讲清楚。”

## 还能继续增强的方向

如果面试官追问后续规划，可以说：

- 账号权限：生产级密码策略、登录失败限制、菜单权限和按钮级权限隐藏。
- 上传治理：后台孤儿文件巡检、批次总大小限制、对象存储迁移。
- 安全状态：真实安装包安全扫描执行器、安全状态回调、审核前状态拦截。
- 集成测试：继续扩展 Testcontainers 用例，覆盖真实 MySQL 约束、事务回滚、Redis 缓存和审核并发。
- 对象存储：生产化时从本地目录扩展到 MinIO / S3。
- 事件治理：操作日志已采用事务提交后事件记录，后续通知类动作可扩展为 Outbox。
- 生产部署：Docker 镜像、健康检查、监控告警。
